# Workshop guide — HubSpot ↔ AWS Partner Central integration

> **Audience**: AWS partners attending a guided workshop. You'll go from a fresh laptop to a working HubSpot card that creates and updates AWS Partner Central opportunities, end-to-end, in roughly **90 minutes**.
>
> **For LLM assistants**: this guide is written so an AI coding assistant (Kiro, Claude, Copilot, etc.) can pick it up mid-flow and help any participant complete the next step. Each step is numbered, every command is a complete copy-pasteable line, and every placeholder uses the convention `<DESCRIPTIVE-NAME>` so it's unambiguous what the participant must replace. Read the [LLM assistant context](#llm-assistant-context) appendix at the bottom before driving a participant.

---

## Workshop outcomes

By the end of the workshop you'll have:

1. An **AWS sandbox account** (or your own) with Partner Central API access in the **Sandbox catalog**.
2. A **HubSpot developer test account** with the connector's UI Extension card installed on the deal record.
3. The CRM stack deployed to your AWS account: 4 Lambdas (Share / Submit / Refresh / Pull), API Gateway, EventBridge rule, and DynamoDB lock table.
4. A working round-trip: edit a deal in HubSpot → click Share → see the opportunity in AWS Partner Central → edit it in AWS → click Refresh → see the change back in HubSpot.

We'll deploy the CRM mode as the core of the workshop (Modules 0-13). Module 14 is an optional add-on that layers the Partner Central Agent card on top of the same setup (~25 min extra). If you only want the Agent card, the standalone path lives in [`docs/workshop-agent.md`](workshop-agent.md) (~30-45 min).

## 1. Enable the AWS Partner Central sandbox

The connector targets the **Sandbox catalog** during the workshop so you can experiment without affecting any real ACE pipeline.

### 1a. Sign in to AWS Partner Central

You need an existing **AWS Partner Network membership** with API access.

1. Open [partnercentral.awspartner.com](https://partnercentral.awspartner.com) and sign in.
2. If you don't yet have AWS Partner Network access at all, you'll need to register your organisation as an AWS Partner first (separate, longer process — outside the scope of this workshop). The workshop facilitator can provide a shared sandbox account if needed.

### 1b. Activate the sandbox for the Selling API

Reference: [Testing in a sandbox for the AWS Partner Central Selling API](https://docs.aws.amazon.com/partner-central/latest/APIReference/testing-sandbox.html).

The sandbox is a separate "catalog" inside Partner Central. You don't "create" it — you just point your IAM principal at `Catalog: "Sandbox"` on every API call. The connector does this automatically. You only need to grant the IAM user permission to call the Sandbox catalog, which we do in step 2.

> **For LLM assistants**: if a participant asks "where do I create the sandbox?", the answer is **the sandbox is implicit** — there's no portal button to click. They get sandbox access by attaching the IAM policy in step 2.

## 2. Create the IAM user for the connector

The connector authenticates to Partner Central using a long-lived IAM access key. We'll create a dedicated IAM user for it.

### 2a. Sign in to your AWS account

Use **the same AWS account that's linked to your AWS Partner Central account**. If you're not sure, ask your AWS partner admin — Partner Central has a 1:1 link with one specific AWS account.

### 2b. Create the IAM user

1. Open the [IAM console → Users → Create user](https://us-east-1.console.aws.amazon.com/iam/home#/users/create).
2. **User name**: `crm-connector-user`
3. **Provide user access to the AWS Management Console**: leave **unchecked**. This user is for API calls only.
4. Click **Next**.

### 2c. Attach the connector policy

1. **Permissions options**: choose **Attach policies directly**.
2. Click **Create policy** (opens a new tab).
3. Click the **JSON** tab and paste:
    ```json
    {
        "Version": "2012-10-17",
        "Statement": [
            {
                "Effect": "Allow",
                "Action": [
                    "partnercentral:CreateOpportunity",
                    "partnercentral:UpdateOpportunity",
                    "partnercentral:ListOpportunities",
                    "partnercentral:GetOpportunity",
                    "partnercentral:GetAwsOpportunitySummary",
                    "partnercentral:ListSolutions",
                    "partnercentral:AssociateOpportunity",
                    "partnercentral:DisassociateOpportunity",
                    "partnercentral:AssignOpportunity",
                    "partnercentral:SubmitOpportunity",
                    "partnercentral:StartEngagementFromOpportunityTask",
                    "partnercentral:ListEngagementFromOpportunityTasks"
                ],
                "Resource": "*",
                "Condition": {
                    "StringEquals": {
                        "partnercentral:Catalog": "Sandbox"
                    }
                }
            }
        ]
    }
    ```
4. Click **Next**.
5. **Policy name**: `PartnerCentralSandboxConnector`. Description: *"Sandbox-only API access for the HubSpot ↔ Partner Central connector"*.
6. Click **Create policy**.
7. Switch back to the user-creation tab. Click the refresh icon next to the policy list, search for `PartnerCentralSandboxConnector`, check the box, click **Next**.
8. Review and click **Create user**.

### 2d. Generate an access key for the user

1. Click into the new `crm-connector-user`.
2. **Security credentials** tab → **Create access key**.
3. **Use case**: *Other* → **Next** → **Create access key**.
4. **Save the Access Key ID and Secret Access Key somewhere safe**. You'll paste them into Secrets Manager in Lab 10. AWS only shows the secret once.

> **Reminder**: the policy above scopes the user to the **Sandbox catalog only** via the IAM Condition. To move to production later, swap the `"Sandbox"` literal for `"AWS"` and re-attach the policy.

## 3. Configure the AWS CLI

The deploy scripts read your AWS identity from the AWS CLI's default credential chain. We need to tell the CLI which account/region to use.

### 3a. Recommended path — named profile

A named profile keeps the workshop credentials isolated from any other AWS work you do.

```bash
aws configure --profile workshop
```

Paste in order when prompted:
- AWS Access Key ID: from step 2d
- AWS Secret Access Key: from step 2d
- Default region name: `us-east-1`
- Default output format: `json`

Then make every shell session that runs deploy commands use this profile:

```bash
# macOS / Linux / Git Bash on Windows
export AWS_PROFILE=workshop
export AWS_REGION=us-east-1
```

> **Windows note**: in Git Bash, `export` works as shown above. If you prefer PowerShell, the equivalent is `$env:AWS_PROFILE = "workshop"` — but reminder, the deploy scripts themselves require Git Bash.

### 3b. Verify

```bash
aws sts get-caller-identity
```

The output should show your AWS account ID and `Arn: ".../crm-connector-user"`. If it shows a different ARN, your profile isn't being picked up — re-run the `export AWS_PROFILE` command in the same terminal you'll deploy from.

## 5. About the auto-generated app credentials (read-ahead reference)

Our deployment uses HubSpot's **project-based private app** model (platform 2025.2). The app, its access token, and its client secret are all auto-generated when you upload the project via `hs project upload` in Lab 8. **You don't create a Private App by hand.** That means the actual credential capture lives in Lab 9 below — Lab 5 is just a reference for what's about to happen.

If you're navigating HubSpot's UI looking for "Private Apps", you'll find an empty list at **Settings → Integrations → Legacy apps** (formerly "Private Apps"). That's expected. Project-based apps live under **Development → Projects** instead.

The credentials we'll need (collected in Lab 9):

| HubSpot field | Goes to AWS Secrets Manager as | What it does |
|---|---|---|
| **Access token** | `HUBSPOT_PRIVATE_APP_TOKEN` | Lambdas use this to read/write deal properties on HubSpot's API. |
| **Client secret** | `HUBSPOT_CLIENT_SECRET` | Lambdas use this to verify the v3 HMAC signature on every inbound `hubspot.fetch` call. |

Scopes are **defined in code** at `hubspot-card/src/app/app-hsmeta.json:config.auth.requiredScopes` and `agent-card/src/app/app-hsmeta.json:config.auth.requiredScopes`. The project ships them on upload — there's no UI checkbox dance.

## 7. Deploy the CRM stack to AWS

This is the longest step (~5–8 minutes) but it's a single command.

### 7a. Run the unified deploy

From the repo root, in **Git Bash on Windows** or your normal terminal on macOS:

```bash
./infra/unified-deploy.sh --mode crm
```

The script accepts these flags:

| Flag | What it does | Default |
|---|---|---|
| `--mode <crm\|agent\|crm-and-agent>` | **Required.** Which stack(s) to deploy. | — |
| `--profile <name>` | AWS CLI profile to use (sets `AWS_PROFILE`). | AWS CLI default |
| `--region <name>` | AWS region (sets `AWS_REGION`). | `us-east-1` |
| `--env-suffix <name>` | Appends a suffix to stack names + every globally-scoped resource (Lambdas, IAM roles, DynamoDB tables, log groups, Secrets Manager secrets) so dev and prod can coexist in one AWS account. Lowercase letters, digits, hyphens; max 16 chars. Empty default keeps canonical names. | `<empty>` |
| `--skip-build` | Reuse existing `dist/*.zip` bundles. Speeds up redeploys when only the CFN template changed. | off |
| `-y`, `--yes` | Skip the per-stack confirmation prompt. | off |
| `-h`, `--help` | Show usage. | — |

Common combinations:

```bash
# Workshop happy path:
./infra/unified-deploy.sh --mode crm --profile workshop

# Workshop, no prompt (CI / scripted):
./infra/unified-deploy.sh --mode crm --profile workshop -y

# Stand up a "dev" environment alongside an existing canonical deploy:
./infra/unified-deploy.sh --mode crm-and-agent --profile workshop --env-suffix dev

# Re-deploy "dev" without rebuilding the Lambda bundles:
./infra/unified-deploy.sh --mode crm-and-agent --profile workshop --env-suffix dev --skip-build -y
```

`--profile` and `--region` also work as env vars (`AWS_PROFILE`, `AWS_REGION`) if you'd rather export them once for the whole shell session.

> **Sharing an account with another deployment?** If a colleague (or another environment) is already running `ace-share-refresh` / `ace-agent` in the same AWS account, use `--env-suffix` so each stack gets its own resource names. See the [Day-2 operations](architecture.md#parallel-environments---env-suffix) section in the architecture doc for the full picture, including the card-side caveat that `app-hsmeta.json` / `config.local.ts` get overwritten on every deploy.

> **Order matters.** Follow the numbered labs below (8 = upload, 9 = capture credentials, 10 = secrets, 11 = properties) in sequence. Don't run `set-secrets.sh` until after you've uploaded the card and captured its credentials (Labs 8-9), because the values it prompts for don't exist until then.

### 7b. Confirm the API URL

```bash
aws cloudformation describe-stacks \
  --stack-name ace-share-refresh \
  --region us-east-1 \
  --query "Stacks[0].Outputs[?OutputKey=='ApiUrl'].OutputValue" \
  --output text \
  --profile workshop
```

> If you deployed with `--env-suffix dev` (or any other suffix), swap `ace-share-refresh` for `ace-share-refresh-<suffix>` here and in every other `--stack-name` reference below.

The output is something like `https://abc123xyz.execute-api.us-east-1.amazonaws.com`. **Save this URL** — you'll reference it as `<CRM-API-URL>` later.

## 9. Capture the HubSpot credentials

Go back to your HubSpot test account: **[app.hubspot.com](https://app.hubspot.com)**.

Now that the project is uploaded, the access token and client secret exist in HubSpot. They live on the **app's** detail page — one level below the project — not on the project itself. Capture both for Lab 10 (Secrets Manager).

### 9a. Drill into the app

1. In HubSpot, find the **Development** icon in the **left navigation bar** (the icon rail; typically at the very bottom). Click it.
2. In the Development panel, click **Projects**.
3. Click into the project **`hubspot-ace-card-v2`** (the one you uploaded in Lab 8b). The project Overview tab opens.
4. Under **Project Components**, click **ACE Share Card** (the top-level app name from `app-hsmeta.json`). This opens the app's detail page, which has its own tab strip: Overview / Auth / Distribution / Settings.

### 9b. Install the app to surface the access token

The access token only exists *after* the app is installed. New projects need this once.

1. On the app detail page, click the **Distribution** tab.
2. Click **Install now** (it offers to install in your current workshop account — that's correct).
3. Review the requested scopes, click **Connect app**.
4. **Heads-up — HubSpot redirects you out of the developer view.** After **Connect app**, you land on **Settings → Integrations → Connected Apps → ACE Share Card** (a consumer-side admin screen showing Overview / Settings / App log). The access token is **NOT** on that page. Navigate back yourself:
    - In the **left navigation bar** (HubSpot's main left-side icon rail), find the **Development** icon — it's typically at the very bottom. Click it.
    - In the Development panel that opens, click **Projects**.
    - Click **`hubspot-ace-card-v2`**.
    - Under **Project Components**, click **ACE Share Card**.
    - Click the **Distribution** tab. Your account now shows as **Installed**, with the access token underneath.
5. Click **Show** on the **Access token** field and copy the revealed value.
6. Save it as `HUBSPOT_PRIVATE_APP_TOKEN` in your notes for Lab 10.

> The access token is re-showable any time. Re-tracing this path (Development → Projects → hubspot-ace-card-v2 → ACE Share Card → Distribution → Show) is the canonical way to retrieve it later.

### 9c. Copy the client secret

1. Still on the app detail page, click the **Auth** tab.
2. Under **Client credentials**, copy the **Client secret**.
3. Save it as `HUBSPOT_CLIENT_SECRET` in your notes for Lab 10.

> The client secret is shown directly. If a previous workshop attempt exposed it (e.g. pasted into chat), click **Rotate** next to the secret to mint a fresh one — old value is invalidated immediately.

## 10. Populate Secrets Manager

The deploy in Lab 7 created a Secrets Manager secret named `crm-connector/ace-share` (or `crm-connector/ace-share-${ENV_SUFFIX}` if you used `--env-suffix`) with an empty JSON blob. We'll populate it with seven keys.

> **Deployed with `--env-suffix dev`?** Pass the same flag to every helper from this point on so they target the suffixed stack / secret. Either `--env-suffix dev` on each command, or `export ENV_SUFFIX=dev` once for the shell session. Mismatched suffixes silently write the secret to the wrong blob, and the Lambda will fail with `MISSING_SECRET` even though the values exist somewhere in Secrets Manager.

### 10a. Run the interactive setter

The simplest path uses `--auto-bounce` so the script populates the secret AND force-restarts the four Lambdas in one go:

```bash
./infra/set-secrets.sh --auto-bounce --profile workshop
# or, if the stack uses --env-suffix dev:
./infra/set-secrets.sh --auto-bounce --profile workshop --env-suffix dev
```

The script prompts for each key with **hidden stdin** (no echo). Have these values ready and paste at each prompt:

| Prompt | Paste this |
|---|---|
| `AWS_ACE_ACCESS_KEY_ID` | the Access Key ID from Lab 2d |
| `AWS_ACE_SECRET_ACCESS_KEY` | the Secret Access Key from Lab 2d |
| `ACE_REGION` | press Enter (defaults to `us-east-1`) |
| `STAGE_MAPPING` | press Enter to accept the default 7-stage mapping for the standard HubSpot pipeline |
| `STAGE_DISPLAY_NAMES` | press Enter to accept the default labels |
| `HUBSPOT_PRIVATE_APP_TOKEN` | the HubSpot Access Token from Lab 9 |
| `HUBSPOT_CLIENT_SECRET` | the HubSpot Client Secret from Lab 9 |

After the last paste:

1. The script writes the merged blob back to Secrets Manager (`Secret updated.`).
2. **`--auto-bounce`** kicks in: it discovers the four Lambda function names from the stack's CFN resources, reads each Lambda's existing env vars, bumps `FORCE_REFRESH=<timestamp>` while preserving everything else, and waits for each update to land before bouncing the next (avoids `ResourceConflictException` from concurrent updates).
3. You'll see `bounced ace-share-ShareLambda-dev` (etc.) printed sequentially with 5-15s pauses, then `All ... updated.`.

> **Skipping `--auto-bounce`?** Without the flag, the script just updates the secret and prints how to bounce the lambdas yourself. The auto path is safer because it preserves any custom env vars the manual snippet would have to know about (`PULL_LOCK_TABLE`, etc.).

> **Pull Lambda**: needs `PULL_LOCK_TABLE` to read the DynamoDB lock table. The auto-bounce path reads it from the live config and preserves it; you don't need to know its value.

## 11. Provision HubSpot custom properties

The CRM card relies on ~30 deal-level custom properties + ~10 picklists (industry, currency, opportunity type, AWS solutions, AWS products, ...). A wrapper script provisions all of them in one call. Each step is idempotent — re-running is safe.

```bash
./scripts/setup-hubspot-properties.sh --profile workshop
```

Substitute `workshop` with whatever AWS CLI profile you've been using.

The wrapper handles its own setup:
- **Python virtualenv**: if `.venv/` doesn't exist, creates it and runs `pip install -e ".[dev]"` automatically. If one exists, just activates it. Already-active venvs are detected and reused.
- **AWS Products CSV**: downloads from `aws-samples/partner-crm-integration-samples` on first run, caches at `/tmp/SampleAWSProducts.csv`.

Steps the wrapper runs:

### 13c. Verify the opportunity via the Partner Central CLI

The Partner Central web console doesn't expose the **Sandbox catalog** — it's API-only. Use the `partnercentral-selling get-opportunity` AWS CLI command (the same call the Refresh Lambda makes under the hood).

```bash
OPPORTUNITY_ID="O1234567890"   # paste the ID from the deal

aws partnercentral-selling get-opportunity \
  --catalog Sandbox \
  --identifier "${OPPORTUNITY_ID}" \
  --profile workshop \
  --region us-east-1 \
  --output json
```

Substitute `workshop` with whatever AWS CLI profile name you're using.

- `Project.Title` — should match the deal name (`Workshop Smoke Test`).
- `LifeCycle.Stage` and `LifeCycle.ReviewStatus` — what HubSpot wrote back to `aws_stage` and `aws_review_status`.

> **The IAM credentials in `crm-connector-user`** (the ones you put into Secrets Manager in Lab 10) **already have permission to call `GetOpportunity`** on the Sandbox catalog — that's how the Refresh Lambda works. If you'd rather not configure those credentials in your local CLI, drop the `--profile` flag and let the call run as whatever profile you've been using for the workshop deploys (the deploy profile typically has full account access).

> **Quick way to dump just the high-level summary** (less noise than the full response):
>
> ```bash aws partnercentral-selling get-opportunity \ --catalog Sandbox --identifier "${OPPORTUNITY_ID}" \ --profile workshop --region us-east-1 \ --query "{title:Project.Title,stage:LifeCycle.Stage,review:LifeCycle.ReviewStatus,company:Customer.Account.CompanyName}" \ --output table ```

### 13d. Click Refresh (manual backfill from AWS → HubSpot)

This step exercises the **AWS → HubSpot** direction. The deployment ships with two paths for AWS-side changes to flow back to HubSpot:

- **Automatic** via EventBridge — when AWS Partner Central is publishing `aws.partnercentral-selling` events into your account's default event bus, the Pull Lambda fires within seconds of any Partner-Central-side change. No clicking needed.
- **Manual** via the **Refresh from AWS Partner Central** button — reads the live AWS state via `GetOpportunity` / `GetAwsOpportunitySummary` and writes back to HubSpot, on demand. This works regardless of EventBridge state and doubles as a backfill when EventBridge isn't available.

> **EventBridge is opt-in.** Many partner accounts haven't enabled the Partner Central event-notification feature yet. The CRM stack's EventBridge rule deploys regardless and stays harmlessly idle until events arrive — so use the **Refresh** button as the canonical AWS → HubSpot path for the workshop. See [`docs/architecture.md` § Request flow — CRM EventBridge auto-pull](architecture.md#request-flow--crm-eventbridge-auto-pull) for the prerequisites partners need to satisfy before Pull starts firing automatically.

Update the opportunity's industry via the API:

```bash
aws partnercentral-selling update-opportunity \
  --catalog Sandbox \
  --identifier "${OPPORTUNITY_ID}" \
  --customer "Account={Industry=Energy}" \
  --last-modified-date "$(aws partnercentral-selling get-opportunity \
       --catalog Sandbox --identifier ${OPPORTUNITY_ID} \
       --profile workshop --region us-east-1 \
       --query LastModifiedDate --output text)" \
  --profile workshop \
  --region us-east-1
```

> The `--last-modified-date` echo is required by ACE's optimistic- concurrency check — the value must match what `GetOpportunity` just returned, otherwise UpdateOpportunity returns `ConflictException`.

In HubSpot, click the card's **Refresh from AWS Partner Central** button. Within seconds, the deal's `ACE Industry` should flip from `Aerospace` to `Energy`.

If both directions worked — Share to AWS, then Refresh from AWS — the workshop is **complete**. 🎉

## Troubleshooting during the workshop

### `Configuration error: missing secrets: HUBSPOT_CLIENT_SECRET`

Either Lab 10 set `HUBSPOT_CLIENT_SECRET` to an empty value, or the lambdas weren't bounced after Lab 10. Hidden-stdin prompts in `set-secrets.sh` swallow blank input as "leave the existing value alone" — if the value never existed in the first place, the key stays absent.

Confirm what's in the blob:

```bash
aws secretsmanager get-secret-value \
  --secret-id crm-connector/ace-share \
  --profile workshop --region us-east-1 \
  --query SecretString --output text | python3 -m json.tool
```

If `HUBSPOT_CLIENT_SECRET` is missing or empty, re-set just that key (paste the value from Lab 9c → Auth tab → Client secret) and bounce the lambdas:

```bash
./infra/set-secrets.sh HUBSPOT_CLIENT_SECRET --profile workshop

# Bounce all 4 lambdas (see Lab 10 for the loop with `aws lambda wait`).
```

Substitute `workshop` with the AWS profile name you've been using.

### `Unexpected response from backend (status 404)`

The card uploaded to HubSpot is calling an API URL that no longer exists. Most common cause: you redeployed the CRMi stack to a different name (e.g. switched from `--mode crm` to a `--env-suffix` variant, or deleted and recreated the stack), so the card source files were rewritten with the new URL — but the **uploaded** build in HubSpot still has the old URL baked in.

Re-upload the card so the live build picks up the new URL:

```bash
cd hubspot-card && hs project upload && cd ..
```

Then **hard-reload the deal page** (Cmd+Shift+R / Ctrl+Shift+R) because HubSpot caches UI Extension builds on the client.

> **HubSpot CLI fails with `Invalid regular expression flags` / `SyntaxError`?** The CLI requires Node 22+. If you have a Node version manager (mise, nvm, asdf) shimming an older Node onto your PATH, prefix the upload with the right Node binary — e.g. `PATH="/opt/homebrew/bin:$PATH" hs project upload` to use Homebrew's Node 24. `node --version` should report `v22.*` or newer.

### `Unexpected response from backend (status 403)`

The card's `permittedUrls.fetch` doesn't match the deployed API URL. Hard-reload the deal page (Cmd+Shift+R / Ctrl+Shift+R). If still failing, re-run `./infra/unified-deploy.sh --mode crm` and then `cd hubspot-card && hs project upload`.

### `Cannot share: solutions`

The deal's `ace_solutions` field is empty AND `ace_other_solution_description` is empty. Pick at least one Solution Offering or fill in the free-text alternative.

### `INVALID_ENUM_VALUE customer.account.address.stateOrRegion`

ACE rejected the deal's State or Region value. For US customers, ACE expects the **full English state name** (`California`, `New York`, `Massachusetts`), not the 2-letter postal abbreviation (`CA`, `NY`, `MA`).

Quickest fix — re-run the deal helper with the full name:

```bash
AWS_PROFILE=workshop python3 scripts/create-test-deal.py \
  --env-suffix dev \
  --name "<your-deal-name>" \
  --state California
```

The script PATCHes the existing deal in place. Click **Share to AWS** again — it should succeed.

> The helper auto-substitutes 2-letter US abbreviations to the full name and prints a warning, but if you set the value by hand on the deal in HubSpot, the value lands as-is. Check the `ACE Customer State or Region` field in the **About this deal** panel — it should read `California`, not `CA`.

### `BUSINESS_VALIDATION_EXCEPTION lifeCycle.targetCloseDate: Target Close Date should be a future date`

The `closedate` is in the past or today. Edit it to be at least tomorrow.

### `aws sts get-caller-identity` shows the wrong account

Either:
- `AWS_PROFILE` isn't set in this terminal (`echo $AWS_PROFILE` should show `workshop`).
- The AWS CLI is reading default credentials from `~/.aws/credentials` instead of the profile you configured. Run `aws configure list` to see what's active.

### Anything else

The full troubleshooting catalogue is in [`docs/architecture.md` § Common errors](architecture.md#common-errors). Search by the verbatim error message you see in the card or in `ace_sync_error` on the deal.

## Workshop teardown

When you're done with the workshop and want to remove every AWS resource:

> If you deployed with `--env-suffix <name>`, swap `ace-share-refresh` below for `ace-share-refresh-<name>` (and `crm-connector/ace-share` for `crm-connector/ace-share-<name>`, etc.). The artifact bucket name does NOT carry the suffix — it's shared across environments. Lambda function names also pick up the suffix: `ace-share-ShareLambda-<name>`, etc.

```bash
# 1. Delete the CFN stack.
aws cloudformation delete-stack --stack-name ace-share-refresh --region us-east-1
aws cloudformation wait stack-delete-complete --stack-name ace-share-refresh --region us-east-1

# 2. Empty and delete the artifact bucket.
ACCOUNT=$(aws sts get-caller-identity --query Account --output text)
aws s3 rm "s3://ace-share-refresh-deploy-${ACCOUNT}-us-east-1" --recursive
aws s3api delete-bucket --bucket "ace-share-refresh-deploy-${ACCOUNT}-us-east-1" --region us-east-1

# 3. Delete the Secrets Manager secret (force, no 7-day window).
aws secretsmanager delete-secret \
  --secret-id crm-connector/ace-share \
  --force-delete-without-recovery \
  --region us-east-1

# 4. Delete CloudWatch log groups.
for group in /aws/lambda/ace-share-{ShareLambda,RefreshLambda,SubmitLambda,PullLambda}; do
  aws logs delete-log-group --log-group-name "$group" --region us-east-1 || true
done

# 5. (Optional) delete the IAM user.
aws iam list-access-keys --user-name crm-connector-user --query "AccessKeyMetadata[].AccessKeyId" --output text | xargs -n1 -I {} aws iam delete-access-key --user-name crm-connector-user --access-key-id {}
aws iam detach-user-policy --user-name crm-connector-user --policy-arn arn:aws:iam::${ACCOUNT}:policy/PartnerCentralSandboxConnector
aws iam delete-policy --policy-arn arn:aws:iam::${ACCOUNT}:policy/PartnerCentralSandboxConnector
aws iam delete-user --user-name crm-connector-user
```

For the HubSpot side: go to **Development → Projects**, delete the **hubspot-ace-card-v2** project (and **hubspot-ace-agent-card** if you also deployed the Agent card in Lab 14). To delete the test account itself: **Development → Testing → Test Accounts → Actions → Delete**.

## LLM assistant context

> This appendix is for an AI coding assistant helping a workshop participant complete the steps above. Read this entire section before issuing instructions or running commands on the participant's behalf.

### Repo invariants

- Repo root path: wherever the participant cloned to. Throughout the guide we assume `cwd` is the repo root unless a step explicitly says `cd <subdir>`.
- The repo expects **POSIX bash**. On Windows, drive every shell command from **Git Bash**.
- All deploy commands respect the env vars `AWS_PROFILE` and `AWS_REGION`. Defaults: AWS CLI default profile and `us-east-1`. If the participant gets unexpected results, check these first via `echo $AWS_PROFILE` and `echo $AWS_REGION`.
- All Python scripts target Python 3.11+. The pyproject.toml `requires-python = ">=3.11"`. Earlier Python WILL fail at install time.
- `npm ci` in `hubspot-card/` and `agent-card/` triggers a postinstall hook (`scripts/prepare-card-config.mjs`) that materialises gitignored config files. Don't manually create or delete those files; just re-run `npm ci` if they're missing.

### Step state inference

A participant's progress can be inferred from the filesystem and AWS state. Use these signals before asking them where they are:

| Signal | Means they completed |
|---|---|
| `aws sts get-caller-identity` succeeds | Lab 3 |
| `aws cloudformation describe-stacks --stack-name ace-share-refresh` succeeds *(swap for `ace-share-refresh-<suffix>` if you used `--env-suffix`)* | Lab 7 |
| HubSpot project `hubspot-ace-card-v2` visible under Development → Projects | Lab 8 |
| HubSpot client secret + access token captured | Lab 9 |
| Secret `crm-connector/ace-share` has all 7 keys: run `aws secretsmanager get-secret-value --secret-id crm-connector/ace-share --query SecretString --output text \| python3 -m json.tool` and count keys | Lab 10 |
| `hubspot-card/dist/` exists and `hubspot-card/src/app/app-hsmeta.json` has a non-placeholder URL | Lab 7 (deploy materialised the URL) |
| Card visible on a HubSpot deal | Lab 12 |

### Common LLM mistakes to avoid

1. **Don't suggest the participant edit `hubspot-card/src/app/app-hsmeta.json` or `config.local.ts` by hand**. Both are auto-written by `infra/unified-deploy.sh` (which calls `infra/deploy.sh` internally). If the URLs are wrong there, re-run the deploy or set `--skip-build` for a faster re-run.
2. **Don't suggest pasting secrets into shell history**. Always use `./infra/set-secrets.sh` (it reads from hidden stdin). If the participant has already pasted a secret in plain text, advise rotating it (HubSpot → Private App → Auth → Rotate; AWS → IAM → access key → deactivate + create new).
3. **Don't switch to the AWS production catalog without explicit instruction**. The workshop runs entirely on `Catalog: "Sandbox"`. The IAM policy in step 2c is sandbox-only; production needs a different policy and a code-side `ACE_CATALOG` flip. Reference the ["Switch from Sandbox to AWS production catalog"](#switch-from-sandbox-to-aws-production-catalog) section above only if the participant explicitly asks.
4. **Don't run `git push` or initialize a new remote without explicit confirmation**. The workshop participant clones a public repo to deploy from; they don't push back upstream.
5. **Don't suggest `aws configure` without `--profile`**. Without a named profile, you mutate the participant's default profile, which may be tied to other unrelated AWS work.
6. **The `main` branch is just the default branch name**. Several deploy scripts and CLI prompts mention `main`; this is git terminology only.

### Debugging order

When something fails, check in this order before suggesting code edits:

1. **AWS identity**: `aws sts get-caller-identity`. Wrong account → wrong terminal → fix `AWS_PROFILE`.
2. **Stack state**: `aws cloudformation describe-stacks --stack-name ace-share-refresh --query "Stacks[0].StackStatus"` (or `ace-share-refresh-<suffix>` if you used `--env-suffix`). If `*_FAILED`, run `aws cloudformation describe-stack-events` and read the most recent `*_FAILED` event.
3. **Lambda logs**: `./infra/tail-logs.sh share` (or `submit`/`refresh`/`pull`). Add `--env-suffix <suffix>` if you used one. The verbatim AWS error message appears in CloudWatch.
4. **Card front-end**: hard-reload the deal page (Cmd+Shift+R / Ctrl+Shift+R) to drop stale cached card builds. Check the deal's `ace_sync_error` property — that's where backend errors land for the card to display.
5. **HubSpot scopes**: any 401 from HubSpot's API means the Private App is missing a scope. Re-check the scope list in `hubspot-card/src/app/app-hsmeta.json` under `config.auth.requiredScopes`.
